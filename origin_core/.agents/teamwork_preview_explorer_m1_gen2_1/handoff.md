# Handoff Report

## Observation
- **`src/node.py` (lines 13-17)**: `receive_traffic(self, source_id, amount)` blindly accumulates `amount` (`self.current_traffic += amount`) without checking for negative, `NaN`, or `inf` values.
- **`src/node.py` (line 29)**: `step(self)` uses a hardcoded absolute threshold: `if self.surprise > 10.0:`.
- **`src/load_generator.py` (lines 18-25)**: `generate(self)` applies a bounds check `traffic = max(0.0, traffic)` on line 19, but subsequently applies `traffic *= self.anomaly_multiplier` on line 23.
- **`src/load_generator.py` (lines 31-36)**: `generate_deterministic(self, step, anomaly_step)` computes `traffic` and applies `self.anomaly_multiplier` but never applies a `max(0.0, traffic)` bounds check.

## Logic Chain
1. **Invalid State Poisoning**: Because `Node.receive_traffic` lacks validation, any negative, `NaN`, or `inf` values generated by the load generator or network will corrupt `self.current_traffic`. This corruption propagates to `self.expected_traffic` because of the Exponential Moving Average update rule in `step()`, permanently poisoning the node's state.
2. **Fixed Surprise Threshold Flaw**: A hardcoded threshold of `10.0` is overly sensitive at scale. For example, if `expected_traffic = 10000`, a normal 0.2% fluctuation (20.0) exceeds 10.0, causing the `Node` to falsely interpret regular variance as a surprise anomaly and take incorrect "spawn" actions.
3. **Negative Traffic from `generate`**: In the stochastic `generate` method, applying the anomaly multiplier *after* the zero-bound check means that if the multiplier is negative (e.g., simulating a sudden drop in load, or due to misconfiguration), it produces negative traffic.
4. **Negative Traffic from `generate_deterministic`**: The lack of any bounds checking in this method allows a negative `base_traffic` or negative `anomaly_multiplier` to propagate negative traffic directly to the node.

## Caveats
- We did not review `network.py` or `main.py` since the failure specifically localized the boundary violations to `node.py` and `load_generator.py`.
- The fix strategy assumes that raising a `ValueError` in `Node.receive_traffic` is the correct protocol for malformed input (NaN, inf, negative) rather than silently clamping to zero or ignoring.
- For the surprise threshold, we assume combining an absolute minimum threshold with a relative threshold (e.g., `threshold = max(absolute_threshold, expected_traffic * relative_threshold)`) is the intended solution to prevent over-sensitivity at both low and high loads.

## Conclusion
To address the robustness issues:
1. **Input Validation (`src/node.py`)**: Import `math` and add validation in `receive_traffic` to ensure `amount >= 0` and is neither `NaN` nor `inf` (e.g., raise `ValueError`).
2. **Relative Surprise (`src/node.py`)**: Update `Node.__init__` to accept `absolute_surprise_threshold` (default 10.0) and `relative_surprise_threshold` (default 0.1). In `step`, calculate the dynamic threshold as `max(self.absolute_surprise_threshold, self.expected_traffic * self.relative_surprise_threshold)` and use it instead of the hardcoded `10.0`.
3. **Bounds Enforcement (`src/load_generator.py`)**: In both `generate` and `generate_deterministic`, compute the final traffic value (including any anomalies) and apply `traffic = max(0.0, traffic)` as the very last step before returning.
